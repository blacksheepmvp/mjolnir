## 🎯 Objective of v0.2.6a

v0.2.6a is a **minimal safety hotfix** whose goals are:

1. **Prevent** users from entering the “bottomless pit” state in Basic-like configurations.
2. **Intercept** any existing invalid Basic+Nothing configs and route users to onboarding instead of rendering a void.
3. **Update gesture defaults** so that, even if a user ends up in a bad or weird state, they have a **reliable gesture-based escape hatch** in Advanced/Complete mode.

No auto-repair, no onboarding rework, no new Activities.

## 🧨 The “Bottomless Pit” Bug (Conceptual)

The “bottomless pit” happens when:

* One screen is assigned a frontend
* The other screen has no real app surface (empty / frozen frame / void)
* Mjolnir is set as the default Home, so the pit is re-entered after reboot
* User cannot reach Settings, launcher chooser, or Mjolnir config
* Device appears soft-locked, forcing factory reset for some users

On this device, `<Nothing>` is not a normal app choice — it is effectively:

> **“Pass home through to Quickstep for this screen.”**

That only makes sense in Advanced/Complete mode with Quickstep as actual system home.

Therefore:

> **Basic-like configurations must never allow `<Nothing>` on either screen.**

## 🟦 Requirement 1 – Remove `<Nothing>` from Basic Mode Configuration

**Goal:** In Basic-like conditions, users cannot select `<Nothing>` at all.

### 1.1. Definition of “Basic-like” conditions (behavioral, not a field)

Treat a configuration / setup flow as Basic-like when:

* Home Tile is disabled / false / 0
* Top Home and Bottom Home both have selected values
* Mjolnir is not yet set as default home
* Advanced/Complete services/permissions are not enabled

You may already have logic that roughly corresponds to this; reuse whatever exists.

### 1.2. UI change

In any screen where users pick frontends under Basic-like conditions:

* **Do NOT show `<Nothing>` as a selectable option.**
* `<Nothing>` must behave like the special entries that require Advanced/Complete mode:

    * If the user wants `<Nothing>`, they must switch to Advanced/Complete first.
    * Basic-like = **frontend/frontend only**.

Do not refactor the picker architecture; just adjust what appears in the list when Basic-like conditions are true.

## 🟧 Requirement 2 – Detect Invalid Basic+Nothing Configs and Route to Onboarding

0.2.6a **does not** attempt automatic repair or Advanced upgrades.

Instead:

> **If an invalid Basic+Nothing config is detected at startup, launch onboarding instead of trying to render that state.**

### 2.1. What is an invalid config (for this hotfix)?

For v0.2.6a, treat as invalid:

* Any config where one screen uses a frontend and the other is `<Nothing>`, **and** the runtime conditions align with Basic-like behavior (as above).

### 2.2. Where detection happens

At the earliest safe point in **HomeActivity’s** startup flow (or equivalent central entrypoint):

* Load the persisted config.
* If it matches the invalid Basic+Nothing combination:

    * **Do NOT launch the configured frontends.**
    * Instead, invoke the existing onboarding flow on the screen that would otherwise have been “Nothing.”

### 2.3. Constraints

* Do not create any new Activities (e.g., no `InvalidConfigurationActivity` yet — that is later 0.2.6 work).
* Do not restructure onboarding — just call the existing entrypoint.
* Do not attempt to flip to Advanced, enable services, or change the system default launcher.
* Do not rewrite or “fix” the stored config in 0.2.6a.

This is **intercept only**: “invalid → onboarding.”

## 🟨 Requirement 4 – Update Gesture Defaults as a Safety Ladder

Even with these protections, unexpected weird states can still happen over time.
The gesture system must have **sensible defaults** that give users a way to rescue themselves, especially in Advanced/Complete mode.

### 4.1. New default gesture mapping

Change the **default** mapping for the home-button gesture actions to:

* **Single tap** → Both Home
* **Double tap** → Top Home
* **Triple tap** → Recent Apps
* **Long press** → Bottom Home

### 4.2. Behavior requirements

* These are **defaults only**.
  Users may still change these mappings in settings as they wish.
* Existing user-customized mappings should **not** be silently overwritten on update (unless the current architecture already has a defined migration path and the user expects it).

    * For fresh installs / resets, use the new defaults.
* The purpose of these defaults is:

    * To give users a reliable way to:

        * trigger Advanced/Complete mode actions
        * move the active frontend to the other screen
        * open recents
        * generally climb out of strange visual/layout states

Do not change how gestures are stored or applied — just change what the default mapping is when nothing has been set yet.
